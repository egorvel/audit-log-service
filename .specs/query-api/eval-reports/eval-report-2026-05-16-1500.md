## 1. Each AC is testable — **PASS**

All 27 ACs reduce to a server-side observation: an HTTP status, a response body field, a DB-state comparison, or a deterministic ordering check. The previous report flagged AC3.2 because its old wording asserted clients are "not able to interpret" the cursor — a claim about client behavior. AC3.2 has been rewritten: *"The cursor shall be opaque: clients shall not be required or expected to interpret its contents. The system makes no guarantees about the cursor's internal format and may evolve it across releases (signaled by the schema version `v`); the only supported client behavior is to pass a previously returned `next_cursor` back to the system unchanged."* The observable parts (round-trip via `next_cursor`, version-bump rejection) are covered by T6's *"Opaque cursor (AC3.2): clients never need to read it — exercised by asserting an arbitrary base64-shaped string round-trips through the codec and an arbitrary JSON poked into a cursor produces 400/422."* The remaining "no guarantees" clause is a contract claim, not a behavior, and so doesn't need a regression test. AC2.2's "two requests … shall return them in the same order" is testable by replaying the same query on a frozen snapshot.

## 2. Tasks have refs and DoD — **PASS**

Every task T1–T7 carries an explicit `**Refs.**` block (citing both ACs and design §s) and a `**DoD.**` block with verifiable bullets. ## 5. Every AC traces to at least one task whose DoD would fail if the AC regressed — **PASS**

Spot-checks: AC1.10 → T6 *"verify the DB rows are byte-identical before and after the GET (read-only)"*; AC2.1/AC2.2 → T4 *"Seed deterministic data including two events sharing a millisecond. Assert returned rows match filter, are ordered by `(timestamp DESC, id DESC)`"* + T6 *"Ordering & ties (AC2.1–AC2.3): seed two events with identical `timestamp`, assert ULID tiebreaker"*; AC2.4 → T6 *"Assert the response timestamp echoes the DB row's microsecond precision (6 fractional digits) unchanged; no Jackson formatter is configured for `Instant` — pass-through is the contract, so a future 'fix' that pins millisecond width would regress AC2.4"*; AC3.4 → T6 *"insert M new events between page 1 and page 2, assert page 2 still pulls from the original snapshot (stability)"*; AC3.5 → T6 *"AC3.5 must cover both asymmetric cases: cursor encoded with a filter (e.g. `actor=u_42`) replayed with that filter absent, and cursor encoded with no filters replayed with a filter added. Both → 422"*; AC3.10 → T6 *"walk to exhaustion via `next_cursor`, assert union equals seeded set, no duplicates, no omissions"*. Every AC has at least one DoD bullet that would visibly fail on regression.

## 6. No contradictions between `requirements.md`, `design.md`, and `tasks.md` — **WEAK**

Most cross-file claims agree (verified): migration approach (INSERT into fresh table + DDL drop/rename) is identical in requirements §Context, design §2.3, and tasks T2; HTTP status mapping (400 for parse, 422 for semantic) is identical across AC1.8/1.9/3.8/3.9, design §1.4, and the T3/T5/T6 test bullets; ordering `(timestamp DESC, id DESC)` is identical in AC2.1, design §3.1, and the T4/T6 DoD assertions; limit defaults/bounds (50/200) match in AC3.7, design §1.2, §5.2, and tasks T5; the ULID library is now pinned at `com.github.f4b6a3:ulid-creator:5.2.3` in T1 with design §2.2 deferring to tasks (per `_delta.md` v1→v2); the `fh` separator in design §3.3 is the ASCII Unit Separator (U+001F, verified by hex-dump) as the prose claims.

One real drift: **`QuerySpec` ownership conflicts between T3 and T5.** T3 documents `CursorCodec`'s signature as *"`String filterHash(QuerySpec filtersOnly)`"* (tasks.md:141), which presupposes `QuerySpec` exists at T3-merge time. T5 then claims to create it: *"`service.QuerySpec` — internal record carrying `actor`, `resource`, `from`, `to`, `cursor` (decoded `Cursor` or null), `limit`"* (tasks.md:215). Per the dependency graph, T5 depends on T3, so T3 ships first — but T3 cannot compile without `QuerySpec`, which contradicts the tasks.md preamble *"Each task is one safe commit: the tree compiles, all tests pass"*. Design §6.2's CursorCodec entry is silent on `filterHash`'s parameter shape, so design itself is consistent — the contradiction is intra-`tasks.md`. Smallest fix: either change T3's `filterHash` signature to take the four filter primitives (`actor`, `resource`, `from`, `to`) so it doesn't need `QuerySpec`, or move `QuerySpec`'s creation into T3 and reduce T5's scope to *"extend QuerySpec to add `cursor` and `limit`"*.

Minor observation (does not affect verdict): the timestamp in design §1.3's response example uses microsecond precision (`"2026-04-17T11:02:14.123456Z"`) while the timestamp in §3.3's cursor example uses millisecond precision (`"2026-04-17T11:02:14.123Z"`). These are illustrative samples for different events, and T6's DoD pins microsecond pass-through as the actual contract, so this is sample-data sloppiness rather than a contract drift — but matching the precision across examples would remove a reader's stumble.

## 8. Every AC is in EARS form — **PASS**

Every AC carries an explicit EARS label and the prose matches the label's grammar:

- Ubiquitous (*"The system shall …"*): AC1.1, AC1.6, AC1.7, AC1.10, AC2.1, AC2.2, AC2.4, AC3.1, AC3.2, AC3.4, AC3.6, AC3.7.
- Event-driven (*"When …, the system shall …"*): AC1.2, AC1.3, AC1.4, AC1.5, AC2.3, AC3.3, AC3.5, AC3.10.
- Unwanted (*"If …, then the system shall …"*): AC1.8, AC1.9, AC3.8a, AC3.8b, AC3.9a, AC3.9b.

No State-driven or Optional ACs are present, which is appropriate for a stateless read endpoint. AC1.7's subject is "Each event" rather than "The system", and AC2.2's subject is "The ordering", but both still match the Ubiquitous *X shall Y* grammar.

Minor note (does not affect verdict, called out by the prior report and unchanged): AC3.7 bundles three claims (limit is optional, default is 50, max is 200). Each is well-formed Ubiquitous prose, but splitting into atomic ACs would make individual regressions easier to attribute. Atomicity is a separate quality dimension from EARS form.
